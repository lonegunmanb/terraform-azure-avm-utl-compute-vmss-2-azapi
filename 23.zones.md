### 2. CustomizeDiff - Partial ForceNew Logic

**Go Code Evidence:**

```go
CustomizeDiff: pluginsdk.CustomDiffWithAll(
	// Removing existing zones is currently not supported for Virtual Machine Scale Sets
	pluginsdk.ForceNewIfChange("zones", func(ctx context.Context, old, new, meta interface{}) bool {
		oldZones := zones.ExpandUntyped(old.(*schema.Set).List())
		newZones := zones.ExpandUntyped(new.(*schema.Set).List())

		for _, ov := range oldZones {
			found := false
			for _, nv := range newZones {
				if ov == nv {
					found = true
					break
				}
			}

			if !found {
				return true  // Force new if any old zone is not in new zones
			}
		}

		return false
	}),
	// ...
),
```

**Behavior:**
- ForceNew = true ONLY when a zone that exists in old state is NOT in new state (zone removal)
- ForceNew = false when adding new zones or keeping existing zones
- This is asymmetric behavior: adding zones is allowed, removing zones forces replacement

**Implementation:**

1. Read existing state using data source:
```hcl
data "azapi_resource" "existing" {
  type                   = "Microsoft.Compute/virtualMachineScaleSets@2024-11-01"
  name                   = var.orchestrated_virtual_machine_scale_set_name
  parent_id              = var.orchestrated_virtual_machine_scale_set_resource_group_id
  response_export_values = ["zones"]
}
```

2. Compute ForceNew trigger:
```hcl
locals {
  existing_zones = try(
    data.azapi_resource.existing.output.zones != null ? toset(data.azapi_resource.existing.output.zones) : toset([]),
    toset([])
  )
  new_zones = var.orchestrated_virtual_machine_scale_set_zones != null ? var.orchestrated_virtual_machine_scale_set_zones : toset([])

  # Check if any existing zone is removed (not present in new zones)
  zones_force_new_trigger = anytrue([
    for zone in local.existing_zones : !contains(local.new_zones, zone)
  ])
}
```

3. Apply in replace_triggers_external_values:
```hcl
replace_triggers_external_values = {
  zones = { value = local.zones_force_new_trigger }
}
```

**Value Design:** The `value` field contains `local.zones_force_new_trigger` (a boolean), which changes from `false` to `true` only when zone removal is detected. This ensures replacement happens if and only if zones are removed, matching the provider's asymmetric ForceNew behavior.

**Exact Replication:** ✅ The implementation exactly matches the provider's CustomizeDiff logic:
- Iterates through old zones
- Checks if each old zone exists in new zones
- Returns true (forces new) if any old zone is missing
- This enables adding zones without replacement while preventing zone removal without replacement

## Critical Review & Edge Cases

### Null Semantics
- **`null` zones:** Means no availability zones specified. The field is omitted from the API request body.
- **Empty set `[]`:** Would be converted to empty list, but the conditional prevents sending it (only sends when `!= null`)

### Boundary Conditions
- **Adding zones (e.g., `["1"]` → `["1", "2"]`):** ✅ Allowed - does NOT force new resource
- **Removing zones (e.g., `["1", "2"]` → `["1"]`):** ❌ Forces new resource
- **Changing zones (e.g., `["1"]` → `["2"]`):** ❌ Forces new resource (zone "1" was removed)
- **Null → zones:** ✅ Allowed - does NOT force new resource
- **Zones → null:** ❌ Forces new resource (all zones removed)

### Idempotency
- **Set to list conversion:** Using `tolist()` on a set is safe - it produces a stable list
- **Order independence:** The logic uses `contains()` which checks membership regardless of order
- **Comparison logic:** Compares zones as sets (using `toset()` and `contains()`), ensuring order doesn't matter

### Safe References
- **Existing state:** Wrapped in `try()` to handle cases where resource doesn't exist yet (create scenario)
- **Null handling:** Both `existing_zones` and `new_zones` default to empty sets to avoid null reference errors
- **Data source:** Uses `response_export_values = ["zones"]` to only fetch needed data

### Edge Case Analysis

**Scenario 1: First creation (no existing resource)**
- `existing_zones` = `[]` (via try fallback)
- `new_zones` = user input or `[]`
- `zones_force_new_trigger` = `false` (no old zones to remove)
- Result: ✅ Resource created successfully

**Scenario 2: Update adding zones**

- `new_zones` = `["1", "2"]`
- Check: Zone "1" exists in new zones? Yes
- `zones_force_new_trigger` = `false`
- Result: ✅ Update in-place

**Scenario 3: Update removing zones**

- `new_zones` = `["1"]`
- Check: Zone "2" exists in new zones? No
- `zones_force_new_trigger` = `true`
- Result: ✅ Forces replacement

**Scenario 4: Update removing all zones**
- `existing_zones` = `["1", "2"]`
- `new_zones` = `[]`
- Check: Zone "1" exists in new zones? No
- `zones_force_new_trigger` = `true`

#### Issue 2: Wrong Value Design

**Problem:**
The executor put the actual zones data in the `value` field instead of the trigger condition.

**Executor's Approach:**

**Why This Violates executor.md:**

From executor.md Section "Understanding `value` in replace_triggers_external_values":
> - **Key Principle**: The `value` should change **if and only if** the condition that requires replacement is met
> - **Common Mistake - Don't do this:**
> ```hcl

> **Correct Approach:**
> ```hcl
