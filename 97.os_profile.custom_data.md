# Task #97: os_profile.custom_data Argument

## Summary

Implemented the `custom_data` argument from the `os_profile` block in `azurerm_orchestrated_virtual_machine_scale_set` to `azapi_resource`. This field contains Base64-encoded custom data for VM initialization and is sensitive. Due to being a sensitive field within a nested block, it requires an independent ephemeral variable with a version tracker.

## Special Handling

### 1. Sensitive Field in Nested Block

**Critical:** Since `custom_data` is a **sensitive field within a nested block** (os_profile), Terraform cannot use `ephemeral = true` within an `optional(type)` object structure. Therefore, an **independent ephemeral variable** is REQUIRED.

**Implementation:**
- Created `migrate_orchestrated_virtual_machine_scale_set_os_profile_custom_data` as an independent ephemeral variable
- Created `migrate_orchestrated_virtual_machine_scale_set_os_profile_custom_data_version` for version tracking
- Set `ephemeral = true` on the variable
- Added to `sensitive_body_version` for proper update tracking

### 2. Validation - Category 1 (Value Constraints)

**Provider Validation:** `validation.StringIsBase64`

**Requirement:** Must be a valid Base64-encoded string.

**Implementation in migrate_variables.tf:**
```hcl
validation {
  condition     = var.migrate_orchestrated_virtual_machine_scale_set_os_profile_custom_data == null || can(base64decode(var.migrate_orchestrated_virtual_machine_scale_set_os_profile_custom_data))
  error_message = "The custom_data must be a valid Base64-encoded string."
}
```

**Rationale:** Replicated the `StringIsBase64` validation from the provider schema. This ensures fast failure at plan time rather than during API calls.

### 3. Null Handling

From the Create method:
```go
customData := ""

// Pass custom data if it is defined in the config file
if v := osProfile["custom_data"]; v != nil {
    customData = v.(string)
}
```

The provider defaults to an empty string if not provided, but the expand functions still set it via `pointer.To(customData)`. In the Azure API:
- If `null` or empty string: No custom data is passed to VMs
- If provided: Base64-decoded and saved as a file on the VM

**Implementation:** When `null`, the entire conditional block is omitted from the merge, resulting in no `customData` property in the API payload.

### 4. Conditional Presence

The field is only added to the API payload when:
1. `os_profile` block is present
2. `custom_data` variable is not null

This is achieved through the nested conditionals in the merge.

### 5. ForceNew

**Analysis:** The schema does NOT have `ForceNew: true`. However, in practice, changing custom_data on an existing VM scale set typically requires recreation of the VMs (not the scale set itself).

**Implementation:** NOT added to `replace_triggers_external_values` since the schema doesn't define it as ForceNew.

## Critical Review & Edge Cases

### Null Semantics

- **`null`:** Field is omitted from API payload (no customData property)
- **Empty string (`""`):** Technically valid Base64, would be sent to API (decoded to empty content)
- **Meaning:** `null` = "no custom data", empty string = "empty custom data file"

### Boundary Conditions

1. **Maximum Length:** Azure API documentation states: "The maximum length of the binary array is 65535 bytes" (after Base64 decoding). The provider does NOT validate this limit - relying on API validation.

2. **Empty String:** Valid Base64 but practically meaningless. Provider allows this.

3. **Invalid Base64:** Caught by validation at plan time.

### Idempotency

✅ **Idempotent:** String comparison is idempotent. Same Base64 string always produces same result.

### Safe References

✅ **Safe:** The field is optional and accessed only after checking that the variable is not null via conditional merge.

### Edge Case Analysis

#### Edge Case 1: Both Windows and Linux Configurations Present

**Scenario:** User provides both `windows_configuration` and `linux_configuration` in the same `os_profile` block.

**Provider Behavior:** The Create method processes them sequentially, with Windows taking precedence (processed first, Linux would overwrite if reached).

**Our Implementation:** Follows the same pattern via nested merge. The `custom_data` is set at the osProfile level, which applies to whichever configuration is used.

**Risk:** Low - This is an invalid configuration that should be caught by user validation.

#### Edge Case 2: Custom Data Size Limit

**Scenario:** User provides valid Base64 string but decoded size exceeds 65535 bytes.

**Provider Behavior:** No validation - defers to Azure API.

**Our Implementation:** No validation - defers to Azure API (same as provider).

**Risk:** Low - Azure API will reject with clear error message.

#### Edge Case 3: Sensitive Data Exposure

**Scenario:** Custom data contains secrets or credentials.

**Provider Handling:** Marked as `Sensitive: true` in schema.

**Our Implementation:** Uses `ephemeral = true` which provides stronger protection than sensitive alone. Terraform will not persist the value in state and will clear it from memory.

**Risk:** Mitigated - Ephemeral provides appropriate security for sensitive initialization data.

#### Edge Case 4: Version Tracking Updates

**Scenario:** User updates `custom_data` value.

**Implementation:** The `sensitive_body_version` with path `properties.virtualMachineProfile.osProfile.customData` will be updated when the user increments the version variable.

**Risk:** Low - Standard pattern for sensitive field tracking.

## Notes

1. **Independent Ephemeral Variable Required:** Because `custom_data` is a sensitive field within a nested block, Terraform's limitations require an independent ephemeral variable. Cannot use `ephemeral` within `optional()` object types.

2. **Applies to Both OS Types:** The `custom_data` field is extracted at the os_profile level and passed to both Windows and Linux expand functions. It applies to whichever OS configuration is actually used.

3. **Base64 Requirement:** Unlike some other custom_data fields in Terraform providers that auto-encode, this field REQUIRES pre-encoded Base64 input.

4. **Version Tracking:** The version variable allows users to force updates to the custom data even when the Base64 string appears unchanged (useful for rotation scenarios).

5. **Empty String vs Null:** There is a semantic difference - null means "no custom data", empty string means "empty custom data file". Our implementation preserves this distinction.

6. **Security Best Practice:** Using `ephemeral = true` provides better security than `sensitive = true` alone, as ephemeral values are not persisted to state and are cleared from memory after use.
